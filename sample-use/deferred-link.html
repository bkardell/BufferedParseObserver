Promised Sources

This is a proposal and prollyfill for "lazily loaded/deferred assets" using BufferedParseObserver.

<link type="text/x-promised"...>

Authors may include <link> tags with the type="text/x-promised" to declaratively indicate a lazy fetch of text early (prior to DOMContentLoaded). Fetched items are not interpreted, but may specify subtypes which allow easy processing upon demand. These may be text/x-promised-script and text/x-promised-json, or author provided extensions (see below).

These tags MUST include a data-src attribute indicating the URL from which the resource should be loaded, and a data-promise attribute indicating a unique promise name.

Authors may optionally set a data-on attribute to signify deferral of the download itself. Valid values are DOMContentLoaded and load. Any unknown value is treated as if no value is present and requests are made ASAP. 

<!-- All of these initiate an async request/promise script source and are not evaluated until you access. -->

<!-- initiate ASAP -->
<link type="text/x-promised-script" 
    data-promise="lazy.evaled.script" 
    data-src="something.js">

<!-- initiate upon DOMContentLoaded -->
<link type="text/x-promised-json" 
    data-promise="lazy.evaled.script2" 
    data-src="something-else.js" 
    data-on="DOMContentLoaded">

<!-- initiate upon load -->
<link type="text/x-promised" 
    data-promise="lazy.evaled.script3" 
    data-src="something-else-still.js" 
    data-on="load">

NamedSourcePromises

The environment exposes an object for dealing with named promises called NamedSourcePromises:

text( promiseId ) Returns the source text of the network request
json( promiseId ) Parses the source text of the network request as JSON and returns
script( promiseId ) eval()'s' the source text of the network request and returns
use( promiseId ) Automatically call the appropriate method (json, script, etc) based on the extended type specified in the link.
ready( callback ) A method indicating that all source promises made during parse have been resolved*.
when( promiseId | [promiseId] ) Returns a standard promise for one or more named promises and auto-interprets values via use().
*Note that because authors may defer via the optional data-on attribute, this may happen comparatively late. Authors are advised to use the more granular when as appropriate.

use(promiseId) and type interpreters

use(promiseId) will attempt to locate a method on the NamedSourcePromises object matching the extended type of the corresponding <link> that made the promise, thus, it is possible for authors to add new type interpreters by simply adding a method to the NamedSourcePromises object. 

<!-- Contrived, but if you shipped handlebars with your page, you might make a compiler type -->
<link type="text/x-promised-handlebars" data-promise="foo" data-src="sometemplate.handlebars">
    
<script>
    /* Automatically pipe x-promised-handlebars through here on use() */
    NamedSourcePromises.handlebars = function (promiseId) {
        return Handlebars.compile( NamedSourcePromises.text(promiseId) );
    };
</script>

<script type="text/x-promise-deferred"...>

Authors may include <script> tags with the type="text/x-promise-deferred" attribute to declaratively indicate code which should be deferred until one or more named promises is resolved. Note that while this shares minor similarities with dependency management, it is not a replacement. Instead, this is about managing the loading and blocking evaluation of resources. Authors are encouraged to use well established or native dependency management mechanisms for full-fledged on-demand management of graphs. These scripts are executed in a context in which they can provide handlers. 

<script type="text/x-promise-deferred" use="my.handlebars.code, my.twitter.data">
    promises.resolved = function (h, twitterData) {
        // wire up some ui...
        var el = document.getElementById("later");
        var target = document.getElementById("target");
        var template = document.getElementById('template');
        el.innerHTML = "Interpret and Apply...";
        el.classList.add("btn-success");
        $(el).on('click', function () {
            var tmpl = Handlebars.compile(template.innerHTML);
            target.innerHTML = tmpl(twitterData);
        });
    }; 
    promises.rejected = function (err) { 
        console.err("ruh roh...rejected\n" + err); 
    };
</script>

A note on the prollyfill being demonstrated here: Functions are written such that authors may provide a window.ProllyfillRoot object which the API will be attacted to for forward compatability. If none is provided, the functions are attached to window. For the demo, I am setting this so that all examples below are attached to the Hitch object.
